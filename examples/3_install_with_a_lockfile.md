# Install with a lockfile

A lockfile is a file generated by TRM to freeze the exact versions of the dependencies used at a specific moment in time.

When a package depends on other packages, those packages might also depend on other packages, and so on. All these versions can change over time. If you install the same package today and then again in six months, the “latest compatible versions” might be different, and that can lead to different behavior or even errors.

A lockfile guarantees that everyone installs the exact same versions, even if newer versions of dependencies exist.

Install without a lockfile:

- TRM picks the newest versions it finds within the declared version range
- Your coworker might get different versions
- Your test system might behave differently than production
- Things break unexpectedly

Install with a lockfile:

- Same versions
- Same dependencies
- Same behavior everywhere

## One version per system

Because of how SAP systems work, there cannot be more than one version of the same package in a single SAP system.
This constraint means that lockfile-based installations can be challenging when multiple packages use the same dependency.

In this case, you must make compromises and perform manual steps, such as choosing a version that works for all dependent packages.

## Generating a lockfile

Lockfiles are useful only for packages with dependencies, so I'll use the `@trmsamples/doc-demo` package we created earlier as an example.

I'll run:

```bash
trm lock <package name>
```

> 💡 Tip: Run this command inside a folder so you can easily locate the generated file.

## Publish a new dependency version

I'll now publish version `1.0.1` of `@trmsamples/doc-utility` (the dependency used by `@trmsamples/doc-demo`).

The reason for publishing `1.0.1` is important to this demo: `@trmsamples/doc-demo` declares `@trmsamples/doc-utility` as a dependency with the version range `^1.0.0`. After I publish `1.0.1`, an end user who installs version `1.0.2` of `@trmsamples/doc-demo` will receive version `1.0.1` of `@trmsamples/doc-utility` because it is the latest version in that range.

## Install

To really replicate the exact state of `@trmsamples/doc-demo` when the lockfile was generated, we need to install it with the lockfile!

Let's run the clean-install command:

```bash
trm clean-install <package name>
```

<div style="display:flex; justify-content:center;">
  <img src="/examples/media/3_5_install_with_a_lockfile.png" alt="Publishing a new release of the package" style="max-width:100%; height:auto; border-radius:8px;">
</div>

TRM installed version `1.0.0` of `@trmsamples/doc-utility` because that version was used when the lockfile was generated.
Behind the scenes, it also verified that the integrity value from the registry matched the value in the lockfile, ensuring that the release was identical.
